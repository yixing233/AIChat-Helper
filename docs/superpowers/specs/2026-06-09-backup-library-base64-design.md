# 备份库页面与 Base64 图片留存设计

日期：2026-06-09

## 背景

扩展已经具备自动备份、独立备份库页面、备份下载与预览能力，但当前备份库仍复用弹窗式列表，页面信息层级较弱；图片文件以 `number[]` 保存二进制内容，体积和可读性都不理想；预览数据也没有稳定保留图片，导致备份库不能可靠呈现历史图片消息。

本设计聚焦两个目标：

- 将备份库改为完整的独立管理页，清楚区分平台、备份记录和预览内容。
- 将导出文件与预览图片改为 base64 留存，下载时可恢复原文件，预览时可离线显示已缓存图片。

## 范围

本次包含：

- 备份文件存储格式从二进制数组迁移到 base64 字符串。
- 兼容旧的 `encoding: "bytes"` 备份记录。
- 为备份记录增加用于预览的 `previewSnapshot`。
- 将图片附件和正文图片尽量缓存为 `data:image/...;base64,...`。
- 备份库页面改成平台导航、备份列表、详情预览三块区域。
- 下载、删除、图片大图预览、加载和空状态交互。

本次不包含：

- 恢复备份库搜索功能。
- 将备份库放回扩展菜单。
- 改动节点导航、节点跳转、批量导出主逻辑。
- 云同步或跨设备备份。

## 数据设计

`StoredBackupFile` 调整为以字符串为核心：

```ts
interface StoredBackupFile {
  path: string;
  mimeType: string;
  content: string;
  encoding: "text" | "base64";
}
```

为了兼容旧记录，读取和下载时继续接受：

```ts
{
  content: number[];
  encoding: "bytes";
}
```

`ConversationBackupRecord` 增加可选预览字段：

```ts
interface ConversationBackupRecord {
  previewSnapshot?: ConversationSnapshot;
  assetStatus?: {
    cachedImages: number;
    failedImages: number;
  };
}
```

职责分离：

- `files` 保存真实导出文件，用于下载恢复。
- `snapshot` 保存原始快照，用于去重和兼容旧逻辑。
- `previewSnapshot` 保存页面预览快照，其中图片 URL 会尽量替换成 data URL。
- `assetStatus` 记录图片缓存结果，供列表和详情页展示。

## 图片留存

生成备份时创建预览快照：

1. 如果图片附件已有 `content`，直接按 `mimeType` 转为 data URL。
2. 如果附件或正文中有远程图片 URL，尝试 `fetch` 图片字节并转成 base64 data URL。
3. 如果远程图片下载失败，保留原 URL，并记录失败数量。
4. 不把非图片远程资源强行内联，避免无意义膨胀。

下载恢复逻辑不依赖 `previewSnapshot`。导出 zip、html、markdown、txt 等文件仍从 `files` 还原，`base64` 会被解码回 `Uint8Array`。

## 页面结构

备份库是独立页面，不再呈现为放大的弹窗。

页面分为四个层级：

- 顶部摘要栏：标题、总备份数、平台数、最新备份时间、估算占用空间。
- 左侧平台导航：全部、ChatGPT、通义千问、豆包、DeepSeek、Claude，每项显示数量和最近备份时间。
- 中间备份列表：展示标题、平台、来源、格式、消息数、文件大小、创建时间、图片缓存状态。
- 右侧预览面板：展示当前选中备份的元信息、下载和删除操作、消息预览、图片缩略图。

默认行为：

- 打开页面后选中最新备份。
- 切换平台后选中该平台最新备份。
- 删除当前备份后选中相邻记录。
- 没有备份时显示全页空状态。
- 某个平台没有备份时显示该平台空状态。

## 视觉与交互

界面风格是安静、清晰的管理台，不使用营销式 hero 或装饰卡片。

设计原则：

- 平台导航使用紧凑侧栏或窄屏横向标签。
- 列表使用分隔线和轻量背景表达层级，避免卡片套卡片。
- 当前选中备份用左侧强调线和浅色背景区分。
- 数字、大小、时间保持对齐，方便扫描。
- 操作按钮使用图标按钮，并提供 `title` 和 `aria-label`。
- 下载时按钮进入加载状态，当前行显示“正在准备文件”。
- 删除前使用确认对话框，避免误删。
- 图片点击打开大图浮层，浮层支持关闭按钮和背景点击关闭。

窄屏布局：

- 平台导航变为顶部横向滚动标签。
- 备份列表和预览上下排列。
- 操作按钮保持可点击尺寸，文本允许换行，不遮挡内容。

## 错误处理

- 读取备份失败：页面显示错误状态和重试按钮。
- 下载失败：当前记录显示错误提示，按钮恢复可用。
- 删除失败：保留记录并显示错误提示。
- 图片缓存失败：预览中保留原 URL，并显示“远程图片未缓存”。
- 旧记录无 `previewSnapshot`：使用 `snapshot` 降级预览，并在详情中标注“旧备份，图片可能依赖原始链接”。

## 测试设计

备份存储测试：

- `Uint8Array` 文件序列化为 `encoding: "base64"`。
- `getBackupExportFiles` 能从 base64 恢复 `Uint8Array`。
- 旧的 `encoding: "bytes"` 记录仍可下载。
- `previewSnapshot` 能保留在记录中。
- 图片附件内容可转为 data URL。

自动备份测试：

- 自动备份生成带 `previewSnapshot` 的记录。
- 图片缓存失败时仍保存备份，并记录失败数量。
- 自动备份去重逻辑不受 `previewSnapshot` 影响。

备份库页面测试：

- 渲染顶部摘要、平台导航、备份列表和详情预览。
- 默认选中新备份，切换平台后更新选中项。
- 图片 data URL 以缩略图展示。
- 下载、删除、大图预览事件正确触发。
- 空状态、旧备份降级提示、加载状态可见。

## 实施顺序

1. 先补备份存储和预览快照测试。
2. 实现 base64 文件序列化、旧 `bytes` 兼容和下载恢复。
3. 实现预览快照图片缓存构建逻辑。
4. 改造自动备份记录创建路径。
5. 重做备份库页面 DOM 结构和交互绑定。
6. 更新样式，完成宽屏和窄屏布局。
7. 运行定向测试、类型检查、完整测试和构建。

## 验收标准

- 新备份的二进制导出文件不再保存为 `number[]`。
- 新备份的图片预览优先使用 base64 data URL。
- 旧备份仍能下载和预览文本内容。
- 备份库是清楚的独立页面，并能按平台管理记录。
- 下载、删除、预览图片都有明确反馈。
- 自动备份失败图片不会阻断整个备份保存。
